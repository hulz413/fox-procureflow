## Context

Fox Procureflow 当前已经具备采购申请、审批、RFQ、PO、收货、发票和三单匹配的 P0 闭环数据。现有前端根页面更像静态采购工作台摘要，部分指标来自前端本地计算或固定展示，无法完整体现集团/公司采购运行状态。

本 change 将根入口升级为 P1「采购看板」：面向管理者、采购负责人和财务人员，提供集团汇总与公司筛选视角。系统仍是集团内部多公司采购平台，供应商池集团共享，但采购申请、审批实例、RFQ、PO、收货、发票和三单匹配结果都属于公司级业务数据；看板必须尊重公司边界，集团视角只能做授权范围内的汇总。

实现继续遵循 roadmap：MySQL 是核心数据源，React + Ant Design + ECharts 构建前端看板，前中期不引入 Redis 指标缓存、RabbitMQ、MongoDB、MinIO、DeepSeek 或新的认证体系。

## Goals / Non-Goals

**Goals:**

- 提供集团和公司两种采购看板视角，展示采购金额、待审批、RFQ、PO、履约、发票和三单匹配异常等聚合指标。
- 暴露只读 dashboard API，基于现有业务表实时聚合，返回摘要指标、趋势、状态分布、供应商分布和异常概览。
- 将现有前端根页面升级为真实后端数据驱动的“采购看板”，并保留公司切换能力。
- 确保公司级看板只读取同一公司的业务数据，集团级看板按当前演示集团下两家公司汇总。
- 准备或复用足够真实的 demo 数据，让看板稳定展示两家公司不同状态和异常风险。

**Non-Goals:**

- 不做采购员个人任务工作台、待办队列聚合或跨模块快捷处理动作。
- 不新增采购申请、审批、RFQ、PO、收货、发票或三单匹配的业务状态流转。
- 不持久化看板快照，不引入 Redis 指标缓存或异步指标预计算。
- 不接入 DeepSeek 或 AI 自然语言经营摘要。
- 不实现 JWT 登录、复杂权限矩阵、真实附件上传、RabbitMQ 事件或外部可观测性组件。

## Decisions

### 1. 看板作为只读聚合能力，不拥有业务状态

新增 `procurement.dashboard` 或相近模块边界，包含 controller、service、DTO 和必要 repository 查询。看板读取现有模块数据，但不修改任何业务表，也不创建新的审批、PO、收货、发票或匹配记录。

聚合输出代表“当前 MySQL 事实数据的管理视图”。申请、审批、RFQ、PO、收货、发票和三单匹配各自的状态流转仍由原模块负责；看板只按当前状态统计。例如待审批来自 `IN_PROGRESS` 审批节点，匹配异常来自当前 `EXCEPTION` 三单匹配结果。

备选方案：在各业务模块页面内分别增加统计卡片。暂不采用，因为 roadmap 需要集团和公司视角的 P1 管理看板，分散统计无法形成管理者入口。

### 2. API 使用显式 scope，避免隐式默认公司

建议 endpoint：

- `GET /api/procurement-dashboard?scope=GROUP`
- `GET /api/procurement-dashboard?scope=COMPANY&companyId=company-digital`

`GROUP` 聚合当前演示集团下两家公司；`COMPANY` 必须传入已知公司 ID。服务层校验公司存在，不允许未知公司回退到 active demo company。响应中返回 `scope`、`companyId`、`companyName`、`generatedAt` 和各类指标区块，便于前端清晰标注数据范围。

备选方案：复用前端 active company 自动决定范围。暂不采用，因为看板的核心价值正是显式比较集团和公司视角，API 需要可测试的范围语义。

### 3. MVP 使用实时 MySQL 聚合，暂不持久化指标

聚合按现有表做同步查询，推荐先用 repository projection 或 service 内组合查询，而不是新增 dashboard 表。MVP demo 数据量小，实时查询足够；读到的是已提交事务后的当前状态，能在用户创建单据或处理异常后通过刷新看板立即反映。

一致性边界：

- 看板 API 只读，不参与上游写事务。
- 上游单据写入成功后，下一次看板查询即可读取最新提交数据。
- 没有 Redis 缓存，因此不存在缓存失效或延迟刷新语义。

备选方案：新增指标快照表或 Redis 缓存。暂不采用，因为 roadmap 明确 Redis 指标缓存是后段性能增强，当前切片更需要低复杂度和可演示闭环。

### 4. 指标定义以可解释、可测试为先

响应建议包含：

- `summary`: 采购总金额、待审批数、进行中 RFQ 数、已发布 PO 数、收货/发票待补齐数、三单匹配异常数。
- `spendTrend`: 按月或按演示月份聚合 PO 已发布金额，使用 PO 币种和含税总额。
- `documentFunnel`: 申请、审批通过、RFQ 可比价、PO 已发布、已收货、已开票、匹配完成的数量。
- `statusDistribution`: 审批、RFQ、PO、收货、发票、三单匹配各状态计数。
- `supplierDistribution`: 按供应商聚合 PO 金额、报价数量或中标 PO 数，供应商基础信息来自集团共享供应商池。
- `exceptionHighlights`: 三单匹配异常、最高严重度、发票金额差异、最近计算时间等风险摘要。

金额使用 `BigDecimal` 聚合并返回币种；MVP demo 默认使用人民币。如果后续出现多币种，先按币种分组返回，不在看板内做汇率换算。

备选方案：一次性定义复杂经营指标，如节省率、采购周期、预算消耗率。暂不采用，因为当前已有业务表还没有完整预算冻结、合同和付款语义，过早定义会制造假精度。

### 5. 前端升级现有根页面为“采购看板”

侧边栏首项从“采购工作台”调整为“采购看板”，继续作为默认首页 `/`，避免新增一个与首页重复的入口。页面加载 dashboard API，而不是从前端静态常量或局部列表推导核心指标。

前端结构建议：

- 顶部范围切换：集团汇总、星河数字科技有限公司、星河智能制造有限公司。
- KPI 区：采购金额、待审批、进行中 RFQ、已发布 PO、匹配异常。
- 图表区：采购金额趋势、单据漏斗、状态分布、供应商分布。
- 风险区：三单匹配异常摘要和可跳转到“三单匹配”页面的入口。

备选方案：新增 `/procurement-dashboard` 并保留 `/` 为采购工作台。暂不采用，因为用户已经区分了看板和工作台，而当前 roadmap 下一步是看板；保留一个静态工作台会加重导航概念负担。

## Risks / Trade-offs

- [Risk] 实时聚合查询跨多个业务模块，service 可能变成报表式耦合点。→ Mitigation: 只读 DTO 和查询集中在 dashboard 模块，不把聚合逻辑反向写入业务服务。
- [Risk] 根页面从“工作台”改为“看板”可能影响演示话术。→ Mitigation: 文案明确是管理看板，执行动作仍留在采购申请、审批、RFQ、PO、收货发票和三单匹配页面。
- [Risk] 采购金额趋势若只基于 PO 可能与申请金额不一致。→ Mitigation: 明确定义“采购金额”为已发布 PO 含税金额；申请金额可作为漏斗或辅助指标。
- [Risk] 多币种未来会让汇总金额失真。→ Mitigation: MVP demo 使用单币种；接口保留 currency 字段，未来按币种分组或接汇率服务。
- [Risk] 现有 `App.tsx` 已较大，继续堆叠看板会降低可维护性。→ Mitigation: 实现时优先抽出 dashboard API/type/helper/page 区块，至少避免新增静态 mock 指标。

## Migration Plan

1. 新增 dashboard 后端 DTO、service、controller 和必要 repository 查询。
2. 必要时新增 seed migration 或调整现有 demo 数据，确保两家公司都有可聚合的采购执行数据、供应商分布和匹配状态。
3. 在 `SecurityConfig` 放行本地 demo dashboard API，同时保持服务层显式 scope/company 校验。
4. 前端新增 dashboard API client/type/query，并将根页面改为后端数据驱动的采购看板。
5. 更新导航、标题、图表和空状态，确保无数据时展示真实空状态而不是静态 mock。
6. 增加后端聚合测试、OpenAPI 测试、前端构建验证和浏览器演示验证。

Rollback: dashboard API 和前端看板为只读能力。若需回滚，可移除导航/根页面改造和 `/api/procurement-dashboard`，不会影响既有采购闭环业务数据。

## Open Questions

- 首页路由是否长期保持 `/`，还是在后续真正建设“采购工作台”时再新增 `/workbench`？当前建议保持 `/` 为采购看板。
- 采购金额趋势是否按 PO 发布日期还是创建日期聚合？当前建议按发布日期或状态记录中的发布动作时间；如果缺少发布时间，则回退到 PO 更新时间。
- 是否需要同时展示英文文案？当前前端已有中英文结构，实现时应同步补齐，避免语言切换后出现旧的静态看板内容。
