## Context

Fox Procureflow 已完成演示主数据、采购申请 Intake 和审批流。当前系统能创建公司级采购申请，提交后按公司、品类和金额生成审批实例，审批通过后停留在审批结果展示阶段，不会创建 RFQ、PO、收货、发票或三单匹配记录。

RFQ 询报价是 P0 核心闭环的下一段：采购员需要把已审批通过的采购申请转换成面向集团共享供应商池的询价记录，录入多家供应商报价，并形成可解释的对比排序，为后续 PO 切片提供候选报价。数据归属继续以 `companyId` 为隔离边界；供应商和采购品类仍是集团共享参考数据；核心交易数据继续存储在 MySQL 中，并通过同步服务调用保持一致性。

## Goals / Non-Goals

**Goals:**

- 新增 RFQ、RFQ 供应商邀请、报价和报价附件元数据占位的 MySQL 表、后端领域模型、REST API 和 Swagger 文档。
- 只允许基于审批状态为 `APPROVED` 的采购申请创建 RFQ，并在创建时快照采购申请头、明细、审批结果和候选供应商。
- 支持采购员从集团共享供应商池选择候选供应商，并校验供应商是否适用于采购申请品类。
- 支持为已邀请供应商录入或更新报价、税率、交期、评分、风险备注和附件文件名/说明等元数据。
- 支持按公司查询 RFQ 列表、详情和报价对比结果，并给出确定性的推荐排序。
- 前端新增“询报价”工作流，覆盖从已审批申请创建 RFQ 到报价录入和对比查看的演示场景。

**Non-Goals:**

- 不创建采购订单，不锁定中标供应商，不发布 PO，也不触发收货、发票或三单匹配。
- 不实现供应商外部门户、邮件/短信通知、供应商自助报价或外部账号体系。
- 不引入 MinIO 真实附件上传；本切片只保存附件意图、文件名、说明和元数据占位。
- 不引入 Redis、RabbitMQ、MongoDB 或异步事件化 RFQ 流程。
- 不接入 DeepSeek 或其他 AI 能力，不做 AI 报价解释或自动决策。
- 不实现真实 JWT 鉴权；继续沿用当前 demo 安全模型，并在服务层校验公司和用户归属。

## Decisions

### 1. RFQ 是独立 P0 交易聚合，引用并快照上游申请

新增 `rfqs`、`rfq_suppliers`、`rfq_quotes` 和 `rfq_quote_attachments` 表。`rfqs` 存储 `rfq_id`、`company_id`、`request_id`、`approval_id`、采购员、状态、标题、币种、目标交付日期、申请头快照和审计时间；`rfq_suppliers` 存储被邀请供应商、邀请状态和供应商风险快照；`rfq_quotes` 存储每家供应商的报价、税额/税率、交期、评分、风险备注和排名相关字段；附件表只存元数据占位。

理由：RFQ 是审批之后的采购执行事实，不能把报价信息塞回采购申请或审批表。快照上游申请和审批结果可以保证后续申请信息变化时，RFQ 对比仍然可审计。

备选方案：只在采购申请上增加 RFQ 字段。暂不采用，因为会混淆申请、审批和采购执行职责，也会让后续 PO 从中标报价生成时缺少清晰的报价聚合。

### 2. 创建 RFQ 时同步校验审批结果并保持事务边界

RFQ 创建在单个数据库事务内完成：

1. 读取采购申请，校验 `companyId` 一致。
2. 读取同一 `requestId` 的审批实例，要求状态为 `APPROVED`。
3. 校验该采购申请尚未创建 RFQ。
4. 校验采购员属于当前公司，供应商来自集团共享供应商池并覆盖申请品类。
5. 创建 RFQ、供应商邀请和申请快照。

状态流转保持小而明确：

```text
APPROVED purchase request
        |
        v
RFQ ISSUED --first quote--> QUOTING --enough valid quotes--> COMPARISON_READY
```

本 change 不提供 `AWARDED` 或 `PO_CREATED` 状态。下一步 PO 切片可以从 `COMPARISON_READY` 的 RFQ 读取报价排序，并定义选择供应商与生成订单的正式状态流转。

备选方案：RFQ 创建后立即进入 `DRAFT`，再单独发布。暂不采用，因为 MVP 演示不包含外部供应商通知，创建动作本身即可表示内部询价已发出。

### 3. 报价录入只允许针对已邀请供应商

报价 API 以 `rfqId + supplierId` 为业务定位，允许采购员录入或更新该供应商的一条当前有效报价。服务层校验 RFQ 公司、采购员公司、供应商邀请关系和金额/交期等字段。每家供应商在同一 RFQ 下只有一条有效报价，更新会覆盖报价主数据并保留更新时间；历史版本不在本切片实现。

理由：MVP 目标是快速演示三家供应商报价对比，不需要完整供应商谈判轮次和版本管理。唯一有效报价让列表、对比和推荐排序更直观。

备选方案：支持多轮报价和报价版本。暂不采用，因为它会引入截止时间、轮次、撤回、历史对比和供应商门户等额外规则。

### 4. 推荐排序采用确定性业务评分而非 AI

报价对比读取 RFQ 下所有有效报价，按确定性评分排序。初始评分可由价格、交期、供应商评分和风险等级组成，例如价格越低、交期越短、供应商评分越高、风险越低排名越靠前；同分时按报价金额、交期和供应商名称稳定排序。API 返回推荐排名和关键比较字段，但不把推荐解释包装成 AI 判断。

理由：roadmap 把“报价推荐解释”放在 P1 AI 范围，P0 只需要可测试、可复现、可演示的排序规则。

备选方案：只按最低价排序。暂不采用，因为演示场景明确需要展示价格、交期、评分和风险，最低价无法体现采购决策的多因素比较。

### 5. API 继续遵守公司隔离和 demo 安全模型

新增 API 示例：

- `GET /api/rfqs?companyId=...&status=...`
- `POST /api/rfqs`
- `GET /api/rfqs/{rfqId}`
- `PUT /api/rfqs/{rfqId}/quotes/{supplierId}`
- `GET /api/rfqs/{rfqId}/comparison`

当前没有 JWT，因此 `SecurityConfig` 临时放行 `/api/rfqs/**` 的本地 demo GET/POST/PUT 调用。服务层仍要求显式传入 `companyId`、`procurementUserId` 或等价 actor 字段，并校验用户、采购申请、审批实例和 RFQ 均属于同一公司。跨公司读取或写入必须返回客户端可见的 4xx 错误。

备选方案：等待认证切片后再做 RFQ。暂不采用，因为 P0 闭环需要先贯通采购执行主流程，且现有切片已经采用服务层归属校验来保护 demo 边界。

### 6. 前端在现有工作台中新增 RFQ 页面

侧边栏新增或启用“询报价”入口，路由为 `/rfqs`。页面使用现有 Ant Design 工作台风格，避免营销式页面。核心区域包括已审批申请选择、RFQ 创建抽屉、供应商多选、RFQ 列表、详情、报价录入抽屉和报价对比表。页面从后端加载主数据、已审批申请和 RFQ 数据，不使用静态 mock。

理由：RFQ 是采购员日常工作台动作，应该与采购申请和审批中心在同一个操作壳内完成。

备选方案：在采购申请详情里直接塞 RFQ 表单。暂不采用，因为 RFQ 会快速发展出独立列表、报价对比和后续 PO 入口，单独页面更清晰。

## Risks / Trade-offs

- [Risk] 一张采购申请只允许一个 RFQ 可能不覆盖真实重新询价场景。→ Mitigation: MVP 先保证闭环清晰，后续可以在 RFQ 上增加轮次或重开机制，而不影响当前单 RFQ 结构。
- [Risk] 报价只保留当前有效版本，无法审计多轮谈判历史。→ Mitigation: 明确多轮报价不在本切片范围，后续可新增 `rfq_quote_versions` 表。
- [Risk] 推荐排序权重可能和真实采购策略不同。→ Mitigation: 使用简单可配置常量或集中函数实现，测试覆盖排序结果，后续 AI 解释或策略配置可以替换评分函数。
- [Risk] 真实附件上传后需要迁移元数据占位。→ Mitigation: 附件表提前保留 `fileName`、`description`、`contentType`、`sizeBytes`、`storageObjectKey` nullable 字段，后续 MinIO 切片只补上传和对象 key。
- [Risk] 当前 demo 安全模型需要前端传 actor 字段。→ Mitigation: 所有写入仍由服务层校验 actor 公司归属，JWT 切片上线后可把 actor 来源替换为 token。
- [Risk] 前端 `App.tsx` 已经较大，继续集中实现会增加维护成本。→ Mitigation: RFQ 实现时优先拆出类型、API 函数和页面级组件，避免把新流程全部堆入主组件。

## Migration Plan

1. 新增 Flyway `V8__create_rfq.sql`，只创建 RFQ 相关新表、唯一约束和公司/申请/状态索引。
2. 新增 RFQ JPA entity、repository、service、controller、DTO 和状态枚举，复用现有异常响应和 API envelope。
3. 扩展采购申请查询能力或新增 RFQ service 内部查询，获取已审批申请和申请明细快照。
4. 扩展 `SecurityConfig` 临时放行 `/api/rfqs/**` demo API。
5. 新增前端 RFQ API、query/mutation hooks、路由、页面、创建抽屉、报价录入和对比视图。
6. 增加后端集成测试、前端构建验证和本地浏览器演示验证。

回滚策略：本 change 只新增 RFQ 表和代码路径。回滚时移除 RFQ API/前端入口并清理 RFQ 表数据，不影响已有主数据、采购申请和审批流。

## Open Questions

- RFQ 编号是否采用 `RFQ-YYYYMMDD-NNNN`，还是要包含公司短码以便演示时更容易识别。
- 推荐排序的初始权重是否需要在 roadmap 或后续管理配置中显式暴露。
- PO 切片是否要从“推荐第一名”直接生成 PO，还是需要 RFQ 先增加人工选定候选报价动作。
