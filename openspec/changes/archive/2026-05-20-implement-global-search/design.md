## Context

Fox Procureflow 当前已经覆盖集团共享供应商池，以及公司级采购申请、审批、RFQ、采购订单、收货、发票、三单匹配和采购看板。右上角搜索按钮位于全局 topbar，但目前只是一个图标按钮，没有 `onClick`、搜索框、后端 API 或结果导航。

这类入口最适合承担“全局业务对象定位器”的角色：用户可能只知道一个 `PR`、`RFQ`、`PO`、发票号、供应商名、审批人或异常线索，需要快速到达对应工作台和详情。由于当前项目仍是 MVP 本地演示范围，搜索应复用 MySQL 中现有业务表和主数据表，保持只读、轻量、可演示。

## Goals / Non-Goals

**Goals:**

- 让右上角搜索成为可用的全局搜索弹窗，而不是装饰性按钮。
- 提供一个只读后端聚合搜索 API，覆盖当前采购闭环中的核心业务对象。
- 明确多公司边界：公司级交易记录按 `companyId` 隔离，集团共享供应商池和采购品类可以作为共享参考数据展示。
- 让搜索结果能跳转到对应工作台，并通过 URL 参数或等价状态打开目标详情。
- 保持搜索不改变采购申请、审批、RFQ、PO、收货、发票、三单匹配或附件状态。

**Non-Goals:**

- 不引入 Elasticsearch、Redis、外部搜索服务、向量库、AI 语义搜索或 DeepSeek 调用。
- 不实现跨集团、多租户或用户自定义权限搜索。
- 不从搜索结果直接执行业务动作，例如审批、发布 PO、收货、开票、处理异常或上传附件。
- 不重写各工作台现有列表筛选和表格结构。

## Decisions

### 1. 新增单一全局搜索 API

新增 `GET /api/global-search?companyId={companyId}&query={keyword}`，返回统一的搜索响应：

- `query`: 规范化后的关键词。
- `companyId`: 当前交易数据检索公司。
- `generatedAt`: 搜索生成时间。
- `groups`: 按结果类型分组的数组。
- 每个 result 包含 `type`、`id`、`title`、`subtitle`、`status`、`companyId`、`companyName`、`supplierName`、`amount`、`currency`、`matchedFields`、`targetPath` 和可选 `targetParams`。

理由：前端只需要调用一个入口，就能得到可展示和可跳转的统一结果；后端集中处理公司隔离和排序，避免前端为了搜索重复拉取多个列表。

备选方案：前端聚合已有列表 API。暂不采用，因为它只能搜索已加载页面数据，无法可靠覆盖收货、发票、三单匹配、审批详情和跨页面对象，也容易绕过公司隔离校验。

### 2. 使用 MySQL LIKE 查询和有限结果集

本 change 使用现有 MySQL 表做同步查询，每类结果限制返回数量，按“精确编号命中优先、近期业务对象优先、异常或待办优先”排序。关键词长度建议至少 2 个字符；空关键词返回空结果和可搜索范围说明，不返回静态业务数据。

理由：当前数据规模是演示级，MySQL 查询足够稳定；不引入 deferred infrastructure，符合 MVP 边界。

备选方案：引入 Elasticsearch 或 Redis 索引。暂不采用，因为这会扩大本地环境复杂度，也超出当前 roadmap 对基础设施的边界。

### 3. 搜索范围按业务价值分层

第一层必须覆盖用户最可能搜索的业务对象：

- 采购申请：`requestId`、标题、状态、申请人、品类、供应商。
- 审批：`approvalId`、关联 `requestId`、标题、当前节点、审批人、状态。
- RFQ：`rfqId`、标题、源申请、供应商、状态。
- 采购订单：`poId`、标题、供应商、源 RFQ、状态。
- 收货/发票：`receiptId`、`invoiceId`、发票号、PO、供应商、登记人。
- 三单匹配：`matchId`、PO、供应商、状态、差异类型和严重度。
- 供应商池：供应商名称、服务范围、地区、风险等级、品类。
- 主数据：公司、部门、用户、预算科目、采购品类。

理由：这些字段覆盖演示人员和采购用户实际记忆线索。结果上下文比“只显示标题”更重要，因为采购业务对象编号多且相似。

### 4. 交易数据默认公司隔离，共享数据明确标识

请求必须携带当前 `companyId`。采购申请、审批、RFQ、PO、收货、发票和三单匹配只查询该公司的记录。供应商池和采购品类作为集团共享参考数据可返回，并在结果中标记为集团共享。公司、部门、用户、预算科目等主数据按现有归属展示：公司级主数据只返回当前公司相关记录，集团级数据可共享。

理由：搜索入口在全局 topbar，但不能变成绕过公司隔离的后门。当前无 JWT，仍以显式 `companyId` 和服务层校验维持 demo 安全模型。

### 5. 前端采用 Command Palette 弹窗

点击搜索按钮或按 `Cmd/Ctrl + K` 打开居中的搜索弹窗。弹窗包含：

- 顶部搜索输入。
- 加载、错误、空结果状态。
- 按类型分组的结果列表。
- 键盘上下选择、回车打开、Esc 关闭。
- 每条结果展示标题、编号、类型、状态、公司、供应商或金额等上下文。

理由：搜索结果类型多，topbar 内联输入空间太小；Command Palette 更适合承载分组结果和键盘快速跳转。

### 6. 结果跳转使用已有路由加目标参数

搜索结果的 `targetPath` 指向现有页面，例如 `/purchase-requests`、`/rfqs`、`/purchase-orders`、`/receipts-invoices`、`/three-way-matching`、`/suppliers` 或 `/master-data`。目标参数使用稳定 query，例如 `?requestId=...`、`?rfqId=...`、`?poId=...`、`?invoiceId=...`、`?matchId=...`、`?supplierId=...`。

对应工作台在加载后识别 query 参数并打开详情抽屉或定位目标行；如果目标不可用，则保留页面并显示空态或错误提示。

理由：这样可以让搜索结果可分享、可刷新，也不需要引入新的全局状态库。

## Risks / Trade-offs

- [Risk] 多个业务表聚合查询可能让搜索服务变成大而杂的模块。→ Mitigation: 使用明确 result builder 和类型枚举，每类结果独立查询、独立限制数量。
- [Risk] 搜索结果跳转到详情需要改造多个工作台的 query 参数读取。→ Mitigation: 先支持核心对象的打开行为，无法直接打开的主数据结果至少跳转到正确页面并保留搜索上下文。
- [Risk] LIKE 查询对中文分词、拼音和错别字不友好。→ Mitigation: 明确 MVP 只支持编号、名称、标题、状态和关键字段的包含匹配；语义搜索后续另开 change。
- [Risk] 搜索接口暴露跨模块摘要，可能不小心返回其他公司交易数据。→ Mitigation: 每个交易查询都必须带 `company_id = :companyId` 条件，并补充集成测试覆盖交叉公司数据。
- [Risk] 搜索弹窗加载失败会影响 topbar 体验。→ Mitigation: 弹窗内展示错误态，关闭弹窗后不影响原工作台导航和操作。

## Migration Plan

1. 新增搜索 DTO、service、controller 和 OpenAPI 注解，不改变现有业务 API。
2. 前端将 topbar 搜索按钮连接到全局搜索弹窗，保留现有 topbar 布局。
3. 为各目标工作台补充 query 参数打开详情的轻量适配。
4. 增加后端集成测试和前端组件测试，验证搜索结果和跳转行为。

回滚策略：移除 `/api/global-search` 后端模块和前端搜索弹窗，搜索按钮可回退为无行为按钮或隐藏；已有采购业务数据和状态不受影响。
