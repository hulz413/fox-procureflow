## Context

现有审批流已经完成审批规则匹配、审批实例、节点、记录、审批中心待办、审批详情抽屉和采购申请详情审批摘要。当前后端模型把路径节点和审计记录拆开：`approval_nodes` 表达计划中的步骤、审批人、状态和激活/完成时间；`approval_records` 表达发起、通过、驳回、撤回等不可变动作事实。

问题主要出在展示层：审批详情同时渲染“审批路径”和“审批记录”两个区域，用户需要自己把“部门负责人审批 / 周明远 / 已通过”和“周明远 / 通过 / 备注 / 时间”对应起来。集团内部采购审批是高频扫描场景，审批人和申请人更需要一个按业务顺序排列的审批进度，而不是两份相互参照的列表。

本 change 不改变公司隔离、用户归属或审批状态流转。所有审批数据仍按 `companyId` 归属，审批操作仍由 service layer 校验当前审批人、申请人和公司上下文。

## Goals / Non-Goals

**Goals:**

- 在审批中心详情抽屉中提供统一“审批进度”视图，合并路径节点和审计记录的展示语义。
- 在采购申请详情中复用同一审批进度语义，避免同一申请在两个工作台里出现不同阅读方式。
- 保持路径完整性：已处理、处理中、待处理、已取消节点都必须可见。
- 保持审计完整性：发起、通过、驳回、撤回记录的操作者、备注和时间戳都必须可见。
- 保持后端 API 和数据库职责清晰，优先用现有 `nodes` 和 `timeline` 派生前端视图模型。

**Non-Goals:**

- 不修改审批规则匹配、节点推进、终态判断或撤回规则。
- 不合并 `approval_nodes` 和 `approval_records` 表，也不删除现有 DTO 字段。
- 不引入并行审批、会签、转交、加签或审批模板编辑能力。
- 不改变 RFQ、PO、收货、发票、三单匹配或 AI 建议的下游边界。

## Decisions

### 1. 展示合并，数据模型保持拆分

前端新增或重命名共享展示组件，例如 `ApprovalProgress`，由 `nodes` 和 `timeline` 组合出可渲染的进度项。后端继续返回现有审批详情结构，除非实现时发现现有字段不足以表达已存在事实。

理由：节点和记录承担不同一致性职责。未来节点可能还没有任何记录，但路径必须提前展示；发起和撤回可能不绑定具体节点，但仍属于审批审计。保留拆分可以避免为了 UI 简化而削弱审计模型。

备选方案：后端新增聚合后的 `progress` 字段。暂不采用，因为现有 API 字段已经足够，前端聚合风险低，也避免对后端契约做不必要扩展。

### 2. 进度项按业务路径优先排序

统一视图的主顺序以审批路径为骨架：

```text
发起事件
  ↓
步骤 1：部门负责人审批（已通过）
  └─ 通过记录：操作者、时间、备注
  ↓
步骤 2：财务审批（处理中）
  ↓
终止事件：撤回 / 驳回完成提示（如存在）
```

组合规则：

- `timeline` 中 `nodeId` 为空的 `CREATED` 记录展示为进度开头的“发起”事件。
- 每个 `node` 形成一个步骤项，按 `stepOrder` 排序，并展示节点名称、审批人和节点状态。
- `timeline` 中带 `nodeId` 的记录挂在对应节点下，展示动作、操作者、时间和备注。
- `WITHDRAWN` 等无节点终止事件展示为进度末尾的独立事件。
- 如果出现无法匹配节点的记录，仍按时间顺序展示为独立审计事件，避免隐藏数据。

备选方案：完全按 `timeline.createdAt` 排序。暂不采用，因为未来待处理节点没有记录，会从视图中消失，无法表达完整审批路径。

### 3. 采购申请详情采用同一组件的完整/紧凑模式

审批中心详情可以展示完整节点、记录、上下文和操作区；采购申请详情在已能加载审批详情时复用同一完整进度组件，在只能拿到审批摘要时使用同一标题和事件样式展示紧凑进度，避免回退成“审批记录”单独列表。

理由：申请人从采购申请入口查看审批时，阅读目标和审批中心一致，都是确认当前状态和历史动作。复用组件能减少文案、样式和状态颜色分叉。

### 4. 状态流转和一致性边界不变

通过、驳回、撤回仍由后端事务更新审批实例、节点、采购申请状态摘要和审批记录。前端只在操作成功后刷新任务列表、审批详情和采购申请详情；它不能用本地派生进度代替后端状态。

多公司边界不变：任务列表仍按 `companyId + approverId` 查询；采购申请详情和审批详情仍只能展示该业务单据归属公司的审批数据；演示角色切换只改变前端查询参数，不改变后端归属校验。

## Risks / Trade-offs

- [Risk] 前端聚合节点和记录时把无节点事件放错位置 → Mitigation: 为 `CREATED`、节点审批记录、`WITHDRAWN` 和孤立记录分别加组件测试。
- [Risk] 紧凑模式只拿到 `timeline`，无法展示未来节点 → Mitigation: 采购申请详情在存在 `approvalId` 时继续加载审批详情；加载前用统一进度标题展示摘要态。
- [Risk] 统一组件信息密度过高 → Mitigation: 视觉上用步骤主行承载路径，记录作为步骤附属信息，保持每步只暴露最关键字段。
- [Risk] 文案改名影响测试选择器或快照 → Mitigation: 更新测试断言为业务语义，不依赖旧标题“审批路径/审批记录”。

## Migration Plan

1. 新增或改造共享审批展示组件，先让审批中心详情使用统一进度。
2. 将采购申请详情切换到同一展示语义，保留加载审批详情失败时的摘要态。
3. 更新中英文文案和 CSS，移除详情页上相邻双卡片的视觉重复。
4. 补充前端测试和必要的后端契约回归测试，确认 API 不需要破坏性调整。

回滚策略：由于后端 schema 和 API 不变，可回滚前端组件调用和文案到原 `ApprovalPath` + `ApprovalTimeline` 展示。

## Open Questions

无。当前决策足以进入实现；若实现中发现审批摘要接口缺少紧凑模式必要字段，优先通过加载完整审批详情解决。
