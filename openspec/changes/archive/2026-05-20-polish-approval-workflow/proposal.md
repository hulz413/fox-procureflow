## Why

当前审批详情把“审批路径”和“审批记录”拆成两个相邻区域展示，用户需要在两块内容之间来回对应节点、审批人、状态和动作记录。审批流已经是 MVP 完成能力，本 change 聚焦演示反馈中的可读性打磨：把审批路径和审批记录合并为一条可扫描的“审批进度”，让审批人、申请人和采购员更快理解当前走到哪一步、谁已经处理、后续还有谁。

## What Changes

- 将审批中心详情抽屉和采购申请详情中的审批展示，从分开的“审批路径”和“审批记录”调整为统一“审批进度”视图。
- 统一视图 MUST 同时表达路径节点、节点状态、当前节点、审批人、发起/通过/驳回/撤回等记录、备注和时间戳。
- 已产生记录的节点 SHOULD 在对应步骤下展示最新处理信息；未处理的后续节点仍需可见，用于表达完整路径。
- 发起、撤回等不绑定具体审批节点的记录 SHOULD 在同一审批进度中作为独立事件展示。
- 保留后端 `approval_nodes` 与 `approval_records` 的职责拆分：节点表达计划和状态，记录表达不可变审计事实；本 change 只改变展示组合和前端视图契约。
- 更新相关规格用语，把“审批路径、当前节点、上下文快照和时间线记录”收敛为“审批进度、上下文快照和审计信息”。

非目标：

- 不改变审批规则匹配、串行节点推进、审批通过/驳回/撤回的状态机。
- 不合并或删除数据库中的审批节点表和审批记录表。
- 不新增 RFQ、PO、附件、AI 风险提示或权限体系能力。
- 不引入 Redis、RabbitMQ、MongoDB、MinIO 或新的前端状态管理依赖。

验收标准：

- 审批中心详情抽屉中只出现一个主审批流区域，用户无需在“路径”和“记录”两块卡片之间对应信息。
- 数字公司两级审批示例能同时展示已通过的部门负责人节点、处理中的财务节点，以及发起和通过记录。
- 采购申请详情中的审批摘要/详情与审批中心保持一致展示语义。
- 现有审批 API 响应结构可继续使用；若前端新增派生视图模型，它必须由现有 `nodes` 和 `timeline` 组合得到。

## Capabilities

### New Capabilities

无。

### Modified Capabilities

- `approval-workflow`: 前端审批详情展示要求从分离的审批路径与时间线，调整为统一审批进度视图，并保持路径节点和审计记录完整可见。
- `purchase-request-intake`: 采购申请详情中的审批摘要展示要求同步使用统一审批进度语义。

## Impact

- 前端：影响审批详情抽屉、采购申请详情审批区、共享审批展示组件、相关样式和中英文文案。
- 后端/API：原则上不需要修改 REST 端点或数据库 schema；如实现中发现缺少展示所需字段，应优先在现有 DTO 中补充非破坏性字段。
- 测试：需要覆盖统一审批进度的渲染、节点与记录合并规则，以及审批操作后列表/详情刷新仍然正确。
- 文档/规格：修改 `approval-workflow` delta spec，明确展示层合并语义与底层节点/记录拆分边界。
