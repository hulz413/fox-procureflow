## Context

`establish-demo-master-data` 已经归档，当前后端通过 `/api/master-data/suppliers` 暴露集团共享供应商池，并包含 roadmap 中 5 家演示供应商、风险等级、状态、服务范围、所在地和覆盖品类。RFQ 创建流程也已经复用这些供应商作为集团级参考数据。

当前前端的问题在体验层：侧边栏“供应商池”没有路由，用户点击无反馈；供应商列表只作为“组织与主数据”页面的一块只读表格出现。对演示者和采购员而言，这会让“支持集团共享供应商池”这个产品边界不够显性。

本 change 应把供应商池补成采购工作台的一等页面，同时保持它的主数据属性：供应商池归集团共享，采购申请、RFQ、PO、收货、发票和三单匹配仍按公司归属隔离。供应商池页面只读，不引入供应商维护、准入流程或任何交易状态变更。

## Goals / Non-Goals

**Goals:**

- 提供独立 `/suppliers` 页面和侧边栏导航入口，让供应商池可直接打开。
- 从真实后端 master data API 获取供应商、品类和组织上下文，不使用前端静态 mock 数据填充列表。
- 支持采购员按关键词、品类、风险等级和状态筛选供应商，并打开详情抽屉查看供应商与品类覆盖。
- 在页面中明确展示“集团共享供应商池”和“公司级采购交易隔离”的边界，支持两家公司共享同一供应商池的演示。
- 保持页面只读，不提供会改变供应商主数据或采购交易状态的操作。

**Non-Goals:**

- 不新增供应商创建、编辑、删除、停用、准入、绩效评分维护或供应商门户。
- 不新增供应商资质附件、真实文件上传、MinIO bucket 或附件下载能力。
- 不改变 RFQ、PO、收货、发票、三单匹配的状态流转和一致性边界。
- 不引入新的数据库表、消息队列、缓存、AI 行为或外部服务依赖。

## Decisions

### 1. 将供应商池实现为独立前端工作台页面

前端新增 `/suppliers` 路由，并在工作台路由判断中加入 `isSupplierPoolRoute`。侧边栏本地化 `navItems` 中的“供应商池”从静态项改为带 `path: '/suppliers'` 的导航项，保持当前图标和数量提示。

理由：供应商池是 roadmap 固定产品边界，独立入口能让演示者不用进入“组织与主数据”页面也能说明集团共享供应商池。相比把“主数据”页面的表格做得更显眼，独立页面更符合采购员工作台的心智。

备选方案：只把静态“供应商池”导航项链接到 `/master-data`。暂不采用，因为它仍然把供应商池埋在主数据验证页里，不能形成独立的采购场景。

### 2. 复用现有只读 master data API，筛选优先在前端完成

页面复用现有 TanStack Query 数据：`/api/master-data/context`、`/api/master-data/suppliers` 和 `/api/master-data/categories`。品类、风险等级、状态和关键词筛选优先在前端完成；如实现时发现品类过滤更适合沿用后端 `categoryId` 查询参数，也只能扩展只读查询行为，不新增写接口。

理由：当前供应商池数据量是演示级主数据，前端筛选简单、响应快、无后端迁移成本，并避免把这个补洞切片扩展成供应商管理模块。

备选方案：新增 `/api/suppliers` 业务 API。暂不采用，因为供应商仍属于集团共享主数据，现有 master data API 已经满足读取与品类过滤。

### 3. 详情抽屉只展示参考数据和边界，不放交易动作

供应商列表行点击或详情按钮打开抽屉。抽屉展示供应商名称、服务范围、所在地、风险等级、状态、共享范围、覆盖品类，以及来自 context 的数据边界说明。抽屉不提供“发起 RFQ”“创建 PO”“编辑供应商”等操作。

理由：当前系统已有 RFQ 页面负责采购交易动作；供应商池页面的职责是查询和理解供应商参考数据。把交易动作放到这里会模糊主数据与交易数据边界，也会引入跨页面状态联动复杂度。

备选方案：在供应商详情中提供跳转 RFQ 的快捷按钮。暂不采用，避免误导用户认为能从供应商直接创建 RFQ；当前 RFQ 仍应从已审批采购申请创建。

### 4. 公司切换只用于边界说明，不改变集团供应商池结果

页面可展示当前选中公司和公司切换控件，或复用工作台当前公司上下文，但供应商列表结果必须保持集团共享一致：切换 `company-digital` 与 `company-manufacturing` 不应过滤掉供应商。公司维度只影响边界说明和演示语境。

理由：这符合现有 spec：供应商池是集团级共享参考数据，公司级隔离只适用于部门、用户、预算科目以及采购交易记录。

备选方案：按公司显示不同供应商池。暂不采用，因为这会改变 roadmap 中“集团共享供应商池”的产品边界。

## Risks / Trade-offs

- [Risk] 页面只读可能让用户期待的供应商维护能力仍然不存在。→ Mitigation: 在标题、副文案和详情边界说明中明确这是集团共享参考数据，并把维护、准入、附件作为非目标。
- [Risk] 前端筛选和后端 `categoryId` 查询并存可能产生实现分叉。→ Mitigation: 页面加载全量 5 家演示供应商作为默认数据源，筛选逻辑集中在一个派生列表中；只有需要减少响应体时才使用后端 `categoryId`。
- [Risk] 供应商池入口加入 dashboard route 判断后，根路由兜底可能误把 `/suppliers` 当作看板。→ Mitigation: `isDashboardRoute` 明确排除 `isSupplierPoolRoute`，并为 `/suppliers` 添加显式 `Route`。
- [Risk] 供应商池页面复用 master data 查询，后端不可用时页面可能显示空表。→ Mitigation: 使用现有 loading/error 状态，空态文案说明主数据暂不可用，不用静态 mock 填充。

## Migration Plan

1. 新增前端路由与导航入口，不涉及数据库迁移。
2. 新增供应商池页面组件，复用现有 API client、类型和本地化文案。
3. 保留“组织与主数据”中的供应商表格，避免破坏已有 master data 验证页。
4. 如实现中修改了 master data 查询参数，补充或更新后端 API 测试和 OpenAPI 验证。
5. 回滚时移除 `/suppliers` 路由和导航 path，原 `/master-data` 供应商表仍可作为兜底展示。

## Open Questions

- 暂无阻塞问题。供应商池在本 change 中按只读工作台处理；供应商维护、准入和资质附件可以作为后续独立切片再讨论。
